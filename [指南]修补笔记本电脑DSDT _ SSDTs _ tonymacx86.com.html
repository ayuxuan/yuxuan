    概述

    为了使许多OS X功能在笔记本电脑上运行良好，您将始终需要正确打补丁的DSDT（也许还有一些SSDT）。本指南的目的是为您的OEM DSDT / SSDT的正确修补提供基础。

    高级用户可能希望通过Clover实现hotpatching。请参阅指南：http : //www.tonymacx86.com/threads/guide-using-clover-to-hotpatch-acpi.200137/

    尽管您可能想从另一台计算机使用DSDT，但几乎总是以失败告终。您无法确定从另一台计算机使用ACPI文件是否有效。即使在硬件配置（BIOS版本，安装的内存量，选择的BIOS选项以及其他硬件差异，如安装了哪个WiFi卡）上的细微差异也会导致差异，如果使用外部ACPI文件，会导致不稳定和怪异的错误。这样的差异可以使不同的OperationRegion地址不同，这使得一个系统的补丁DSDT与另一个系统不兼容。在不同主板的不同运行中生产相同的笔记本电脑型号，以及可能不兼容的ACPI文件也是不常见的。

    请记住，即使您对自己的系统（BIOS，硬件等）所做的更改都需要重新提取，重新打包。

    如果任一下列的改变时，你必须重新提取物，再补丁，因为这些变化可能导致本机ACPI（特别是SystemMemory地区）显著的变化：
    -更新BIOS
    -更改任何BIOS选项
    -改变硬件或内存配置

    的修补过程包括以下几个步骤：
    - 提取本机文件
    - 反汇编本机文件
    - 分析本机文件
    - 修补
    - 保存（编译）和安装


    提取本地ACPI文件

    所有的BIOS实现都向操作系统提供ACPI文件。所以，在任何操作系统上，您都可以提取它们以便稍后进行修补。因此可以在Linux，OS X，Windows甚至Clover引导程序中进行提取。提取的本机文件通常是相同的，尽管由于用于提取的软件，它们可能以不同的方式命名。

    本指南将重点介绍三种提取方法：在Clover中使用F4，在OS X中使用patchmatic或使用Linux。
    只有第一个是推荐的方法。其他人仅供您参考。

    使用Clover F4进行提取（推荐）：建议

    使用Clover F4进行提取，因为易于提取，并且易于比较ACPI / origin和ACPI / patch（用于故障排除）。

    在Clover主引导程序屏幕上，您可以按F4，Clover会将本机ACPI文件转储到EFI / Clover / ACPI / origin。您可以在启动OS X之后访问它们以进行反汇编和修补。请注意，一些BIOS的实现与F4的Fn + F4的功能相反，所以如有疑问，请同时按下Fn + F4和F4。转储过程中或转储后没有任何反馈，只是在写入文件时略微延迟。如果写入USB，则延迟更明显，例如从Clover USB启动时的情况。

    有时，Clover F4将会写入重复的SSDT。这些重复将在拆卸过程中造成问题。如果在反汇编过程中遇到问题（重复定义），则需要分析所有SSDT以消除重复的文件。通过查看文件大小很容易看出哪些是重复的。大小相同的文件可能是重复的。

    您可以在终端中查看所有SSDT的文件大小（以字节为单位）：
    代码（文本）：

    ls -l SSDT * .aml
     

    使用“patchmatic”（不推荐）进行提取：

    如果您已经安装了OS X，前提是您目前没有使用任何已修补的ACPI文件进行引导，则可以使用patchmatic提取原生DSDT / SSDT。在这里下载补丁二进制文件：https：//github.com/RehabMan/OS-X-MaciASL-patchmatic（一定要阅读自述文件，因为下载位置是链接的）。为了便于在终端中使用，您应该将二进制文件（在ZIP中）复制到/ usr / bin。

    在安装修补程序之后，可以在终端中调用它，如下所示：

    代码（文本）：

    cd〜 /桌面
    mkdir提取
    cd提取
    patchmatic -extract
     
    补丁二进制文件将提取所有加载的ACPI文件并将其放置在当前目录中。如果您在引导加载程序中使用了影响注入的DSDT / SSDT的任何选项，则不会获得本机ACPI文件，因此请确保您没有使用这些选项。例如，如果使用（Chameleon）DropSSDT = Yes或（Clover）DropOem = true，则在OS X可以加载它们之前将丢弃本机SSDT，因此在patchmatic输出中会丢失它们。对于任何Clover DSDT“修复”也是如此 - 这些修补程序正在修补本机DSDT，并且执行您自己的DSDT修补程序，您不希望这样做。诸如GeneratePStates / GenerateCStates = Yes或者使用Clover / ACPI / SSDT / Generate / CStates / ACPI / SSDT / Generate / PStates的选项将会注入额外的SSDT，这可能会导致拆卸复杂化。

    正是基于这些原因，通过Linux或使用Clover F4进行提取通常比较容易。

    注意：使用'patchmatic -extract'来确认您正在修补DSDT / SSDT，因为您期望这是一个有用的诊断工具。


    使用Linux进行提取（不推荐）：

    在Linux中，可以直接从文件系统获得本地ACPI文件。你可以在/ sys / firmware / acpi / tables和/ sys / firmware / acpi / tables / dynamic找到它们。可以在终端中使用单个命令复制整个集合。

    没有必要安装Linux。只需从USB运行它：http : //www.ubuntu.com/download/desktop/create-a-usb-stick-on-windows。

    在Linux终端中：
    代码（文本）：

    ＃用FAT32格式化的U盘挂载点代替DEST
    sudo cp -R / sys / firmware / acpi / tables DEST
     
    您应该将文件复制到FAT32格式的USB。使用FAT32避免了权限问题，因为FAT32没有文件权限。自动安装的USB的DEST值取决于您使用的Linux版本以及如何启动它。您可以通过在终端中键入“mount”来查看安装点，或者将鼠标悬停在Linux文件资源管理器的卷名上。


    准备反汇编工具

    要正确反汇编解压缩的文件，您需要从终端运行的iasl编译器。

    您将需要最近的iasl版本才能正确拆卸它们。这里有一个适当的版本：https：//bitbucket.org/RehabMan/acpica/downloads/。将iasl二进制文件复制到你的路径（例如/ usr / bin）是一个好主意，所以很容易从终端访问。

    例如，如果您将其下载到〜/ Downloads / iasl.zip，则可以在终端中提取并复制它：
    代码（文本）：


    cd〜 /下载解压缩iasl.zip
    sudo cp iasl / usr / bin
     

    从github构建最新的iasl

    你也可以从我的github构建我的iasl的最新版本。iasl的最新版本最终将在bitbucket链接上提供，但对于那些想要处于“尖端”的用户，您可以自己构建。最新的版本总是倾向于有实验性的和没有经过良好测试的代码。

    假设你已经安装了Xcode：
    代码（文本）：


    mkdir〜 / Projects && cd〜/ Projects git clone https://github.com/RehabMan/Intel-iasl.git iasl.git
    cd iasl.git
     
    然后建立它：
    代码（文本）：

    使
     
    在这一点上，你可以安装它：
    代码（文本）：

    sudo make install
     
    假设您已将MaciASL.app安装到/ Applications，那么您可以在MaciASL中使用新版本（您刚刚构建并安装到/ usr / bin）：
    代码（文本）：

    sudo cp / usr / bin / iasl /Applications/MaciASL.app/Contents/MacOS/iasl61
     


    反汇编ACPI文件

    虽然提取的本机文件可以直接在MaciASL中打开，但不推荐使用。直接在MaciASL中打开一个AML文件将导致MaciASL独立的反汇编该文件（如果是iasl），如果AML对其他AML有复杂的引用，它将不会正确地反汇编。你会留下很多难以解决的错误。

    因此，最好在终端中使用iasl将所有文件作为一个组进行反汇编。要准备，将所有DSDT和SSDT文件放在一个目录中（不要复制不以DSDT或SSDT开头的ACPI文件），并更改名称使其具有.aml扩展名。

    然后在OS X Terminal中反汇编：
    代码（文本）：

    cd“到您放置所有SSDT / DSDT的目录”
    iasl -da -dl DSDT.aml SSDT * .aml
     
    注意：请勿尝试使用-da选项反汇编其他ACPI文件。不起作用。

    注意：请阅读下面有关refs.txt的部分。使用refs.txt需要更多的努力，但可以消除许多常见的错误。

    从这一点开始，您将使用MaciASL 专门处理生成的* .dsl文件。当然，要使用它们，你必须保存为“ACPI机器语言二进制”，扩展名为.aml，并将它们放在引导加载程序将要加载的地方。但是保留修补的.dsl文件，以防将来需要应用更多的修补程序。

    让我简单地陈述它（因为这会出现很多）：如果您直接在MaciASL 中打开一个AML文件，然后单击Compile，你做错了。让它浸泡在你耳朵之间的灰色物质上一分钟。

    注意：使用ACPI 6.1的新工具在处理使用新版本iasl编译的AML文件时更为稳健可靠。ACPI 6.1为编译器添加了一个功能，将外部参考的操作码添加到AML二进制文件中。ACPI解释器会忽略这些数据，但这些数据对于反汇编程序（也仅是iasl的ACPI 6.1版本）非常有用，可以从独立的AML中创建更好的反汇编。因此，您可能会发现使用最新工具重新编译的AML文件可能会更直接地打开。当然，现在的OEM ACPI DSDT和SSDT目前还没有使用新的工具，所以如本指南中所述，您仍然必须先用所有带有选项-da的DSDT / SSDT进行反汇编。

    关于Snow Leopard ACPI实现的注意事项：不幸的是，10.6.8 ACPI已经足够陈旧，以致于与外部操作码一起在AML上窒息。如果您计划在Snow Leopard中使用您的ACPI文件，请在编译AML文件时使用未记录的“-oe”切换到iasl。当你从MaciASL编译（另存为）时，这个选项没有设置，所以你需要在终端中编译你的文件。“-oe”选项禁止在输出AML文件中生成外部操作码。


    用refs.txt反汇编

    有时还有额外的未解决的外部（符号没有在任何文件中定义）。反编译器会试图猜测一些参数，但通常猜测不好。您可以通过在文本文件中提供外部声明来更正它。一些常见的未解决的符号是SGPO，ECRD，ECWT和MMTB。

    下面的refs.txt内容有一些常见的（不是很常见的）缺少的符号（正如这个线程中的用户所报告的）反汇编器往往被混淆了。

    首先在您的DSDT / SSDT文件所在的目录中创建refs.txt：
    代码（文本）：

    外部（MDBG，MethodObj，1）
    外部（_GPE.MMTB，MethodObj，0）
    的外部（_SB.PCI0.LPCB.H_EC.ECWT，MethodObj，2）
    外部（_SB.PCI0.LPCB.H_EC.ECRD，MethodObj，1）
    External（_SB.PCI0.LPCB.H_EC.ECMD，MethodObj，1）
    External（_SB.PCI0.PEG0.PEGP.SGPO，MethodObj，2）
    External（_SB.PCI0.GFX0.DD02._BCM，MethodObj，1）
    External _SB.PCI0.SAT0.SDSM，MethodObj，4）
    外部（_GPE.VHOV，MethodObj，3）
    外部（_SB.PCI0.XHC.RHUB.TPLD，MethodObj，2）
     
    注意：创建refs.txt的一个方便的方法是在终端中使用pbpaste。复制上面的文本到剪贴板（我假设你知道如何做到这一点），然后：
    代码（文本）：

    pbpaste> refs.txt
     
    这将在您当前的工作目录中创建refs.txt。

    然后在拆卸过程中使用它：
    代码（文本）：

    iasl -da -dl -fe refs.txt DSDT.aml SSDT * .aml
     
    分析本地ACPI

    反汇编之后，您可能需要查看每个.dsl文件的内容，以熟悉每个.dsl文件中的内容。某些修补程序取决于内容。例如，如果打补丁来禁用独立图形设备，则可能正在寻找与该设备相关的_OFF方法（此过程将在本指南后面介绍的单独指南中介绍）。

    除修补以禁用独立图形设备外，不需要修补任何SSDT，因为通用重命名已经由Clover笔记本电脑指南（https://www.tonymacx86.com/threads/guide-boot -the-OS-X的安装程序上的笔记本电脑与- clover.148093 /）。重命名最好通过config.plist / ACPI / DSDT / Patches来完成，因为它可以避免与修补手动重命名相比的许多错误。

    大多数情况下，您应该只关注DSDT.aml。


    修复错误

    即使通过一次反汇编（iasl与-da选项），本机文件仍然可能有错误。由于iasl随着时间的变化，iasl本身的缺陷以及笔记本电脑和OEM之间的编译环境的差异，反汇编的文件有错误。例如，错误（我的理论）的一个常见原因是，引用的一些方法实际上是在Windows内部（例如，MMTB和MDBG）。也有明显的情况，代码中的错误或者代码是非单调写入的（有时难以区分）。

    所以..在确定你需要哪些文件之后，你必须对它们进行补丁，以便它们无错地编译。MaciASL的笔记本补丁存储库中有许多这种错误的常见补丁。

    MaciASL：https：//github.com/RehabMan/OS-X-MaciASL-patchmatic
    笔记本补丁：https：//github.com/RehabMan/Laptop-DSDT-Patch

    注意：我不用DSDT编辑器测试我的补丁。它有太多的bug和iasl的一个非常旧的版本。请不要问我这件事。

    确保始终阅读自述文件，以便从正确的位置下载并正确设置MaciASL。语法/错误问题的补丁在名称中以“[syn]”开头。较老的DSDT常见的例子是“Fix _PLD Buffer / Package Error”，“Fix TNOT Error”和“Fix FPED Parse Error”。为了确定你需要哪个补丁，你可以看看来自iasl编译器的错误消息和错误被检测到的代码。您也可以尝试应用修补程序，以查看是否进行了更改，如MaciASL的“预览”窗口中所示。如果您不熟悉每种类型的错误，则可能需要进行一些试验和试验/错误。

    对于一些错误，您可以简单地删除导致错误的代码行。但是，这取决于该行是否是正确的代码操作所必需的。例如，通常可以删除由“外部”声明引起的错误以修复错误。如果你愿意，你可以创建自己的自动补丁来删除这些行。

    它有助于有一些ACPI规范和一些编程经验的经验。

    你的目标是让每个.dsl文件无误地编译（警告/评论/优化是可以的）。一旦你有没有错误编译的文件，你可以继续补丁，以解决您可能与您的OS X安装问题。

    注意：最近DSDT中的一个常见错误是ECRW（在_CRS方法中）。这是iasl错误导致的一个非常常见的错误。我不打算为它添加一个MaciASL补丁，因为它最终将由英特尔（这是一个回归）修复。

    常见的补丁

    一般来说，DSDT补丁只能在找到需要的特定补丁后才能应用。但是有几个补丁是通常需要的，并且只有很小的机会导致问题。他们在我的笔记本电脑回购，这里列出：
    “修复_WAK为arg0 V2”
    “HPET修复”
    “SMBUS修复”
    “IRQ修复”
    “RTC修正”
    “操作系统检查修复”
    “非零同步级别修复互斥体”
    “修复PNOT / PPNT“（只在丢弃与CPU相关的SSDT时使用）
    ”添加IMEI“（如果DSDT或SSDT已经有IMEI / HECI / MEI设备

    ，请勿使用）注意：您使用的OS Check Fix补丁无关随着Windows笔记本电脑的版本，

    注意：如果您包含所有OEM SSDT，请勿使用“修复PNOT / PPNT”修补程序。它仅适用于您省略与OEM CPU相关的SSDT的情况。


    这些补丁可用于为USB注入电源属性：
    “6系列USB”
    “7系列/ 8系列USB”

    _PRW补丁可以用来修复“即时唤醒”，即在睡眠开始后的几秒钟内，笔记本电脑不会睡眠。与DSDT中的现有代码相关（也请注意Skylake及更高版本的这些补丁的特定版本），请仅使用“USB _PRW 0x6D（即时唤醒）”或“USB _PRW 0x0D（即时唤醒）”。您应该检查您的DSDT，以确定相关的_PRW方法返回以确定该补丁适合您的DSDT。还在回购中提供了“USB _PRW 0x0D（即时唤醒）”（0x0D和0x6D都是从_PRW返回的XHC / EHC / HDEF的常用值）。

    如果您有Haswell CPU / 8系列芯片组，并且AppleLPC.kext未加载，则应使用此修补程序注入兼容的ID，以便加载：
    “Haswell LPC”

    如果您有Skylake CPU / 100系列芯片组，并且AppleLPC.kext未加载，则应使用此修补程序注入兼容的ID，以便加载：
    “Skylake LPC”
    注意：AppleLPC可能不需要Skylake和后来。

    关于重命名的注意事项：重命名必须是“平衡的”。通常重命名对象以更好地匹配OS X期望的内容（例如，将GFX0重命名为IGPU以实现正确的IGPU电源管理）。在这种情况下，所有引用该名称的DSDT / SSDT也必须重命名。这就是为什么使用config.plist / ACPI / DSDT / Patches更好地完成重命名的原因。

    有关重复标识符的注意事项：您必须确保修补后的文件不包含重复的标识符。一个常见的情况是将_DSM方法添加到一个SSDT中的给定路径，OEM已经在另一个SSDT的相同路径中定义了一个_DSM。为避免此问题，可以使用“删除_DSM方法”修补程序作为您对所有DSDT / SSDT执行的第一个修补程序之一。此外，“重命名_DSM方法到XDSM”是一种替代方案（有时候“删除_DSM方法”暴露了MaciASL中的一个错误）。

    保存由引导加载程序加载的文件

    为了使用修补过的DSDT / SSDT，您必须将它们保存在引导加载程序可以加载它们的位置。每个引导程序位置都是唯一的，对命名有不同的要求。文件必须保存在“ACPI机器语言二进制”（MaciASL->另存为）。保存带有AML扩展名的文本文件（dsl）可能会导致OS X出现恐慌或非常奇怪的行为。

    三叶草：文件应放置在Clover引导加载程序分区（通常是EFI分区），在EFI / Clover / ACPI / patched中。DSDT.aml（如果有）将自动替换OEM DSDT。本指南（以及本指南链接的其他指南）假设您使用config.plist / ACPI / AutoMerge = true，config.plist / ACPI / SSDT / DropOem = false。在AutoMerge = true的情况下，修补后的SSDT可以放置在ACPI中，并且使用它们的原始名称（从ACPI /打补丁）进行修补，并且它们将被插入，这样SSDT的原始顺序不会受到干扰。不需要（或者建议）在AutoMerge = true的情况下使用SortedOrder。其他配置在“推荐的配置”中进行了介绍。

    如上所述，Clover的RehabMan分支中的一个新功能允许您在不使用DropOem = true的情况下替换OEM SSDT，而不使用SortedOrder，同时保持由Clover注入的未修补和修补的SSDT的原始顺序。通过设置config.plist / ACPI / AutoMerge = true，启用此功能。置于ACPI /补丁中时，SSDT必须保留其原始编号方案。此功能在2017年12月15日，Clover_v2.4k_r4359.RM-4506.c5fc0346.zip上完整运行，位于RehabMan bitbucket站点上。从r4334开始，在官方Clover（在sourceforge上）中实现了所需的更改（但是可能会遇到一些bug，请使用RehabMan版本）。主笔记本电脑指南链接的每个plist默认使用AutoMerge = true。

    浮动区域

    在ACPI中，OperationRegion可以定义MMIO区域，SystemMemory区域，EmbeddedControl区域，等等。这些区域通常具有固定的地址，仅取决于机器配置，BIOS版本或BIOS选项。有时，这些地区可能随机或意外地改变。这被称为“浮动区域”。

    由于通过修补DSDT和/或SSDT，我们在给定的时间点提供这些地址的快照，如果BIOS决定将这些区域放置在不同的地址，它们可能不匹配。如果是这种情况，您可能会注意到，某些功能是间歇性工作的，或其他稳定性问题似乎是随机的。

    如果你有随机的浮动区域，你可以尝试Clover的FixRegions功能（config.plist / ACPI / DSDT / Fixes / FixRegions = true）。您可以在Clover Wiki中找到详细信息。注意：只有DSDT中的浮动区域才能由FixRegion修复。SSDT中的浮动区域是有问题的，并且没有好的解决方案，除了不为随机浮动区域提供SSDT的修补SSDT。在补丁SSDT中解决浮动区域问题已经超出了本指南的范围。请注意FixRegions是相对错误的。它不能修复所有区域，有时可能会错误地修复区域。
